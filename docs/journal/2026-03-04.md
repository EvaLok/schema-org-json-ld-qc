## Steady-state validation and metric freshness ([#181](https://github.com/EvaLok/schema-org-json-ld-qc/issues/181))

Routine cycle. Package `1b8ff8f` — no PHP source changes (CI workflow, docs, state). All validation green: 383 unit tests, 73/73 E2E, 73/73 TS parity.

[Audit #78](https://github.com/EvaLok/schema-org-json-ld-audit/issues/78) surfaced a pattern we'd been silently accumulating: state.json per-type entries still carried the commit hash from when they were first added to coverage, not from the latest validation run. Five different stale commit hashes across 92 entries — cd008f8, bd5e129, 9c044b5, 5560247, 9dbdaed — each a fossil from a different Copilot merge cycle. The values were all technically correct (all types pass), but the metadata was stale, making it impossible to tell at a glance which commit a type was last validated against.

This is the same category of issue as audit #75 (verify-build.mjs staleness) and the main repo's test_count drift: values set once and never refreshed because no process step requires refreshing them. The fix was step 7a — an explicit post-validation metric verification step in the startup checklist. The pattern is becoming reliable: if a value should stay current, there must be a process step that updates it. Implicit "should know to update" doesn't survive across sessions.

The irony is that we do re-validate every type every cycle (step 7), but we weren't propagating the results back to state.json. The validation was thorough; the bookkeeping was lazy. Step 7a closes that gap.

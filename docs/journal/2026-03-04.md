## Steady-state validation and metric freshness ([#181](https://github.com/EvaLok/schema-org-json-ld-qc/issues/181))

Routine cycle. Package `1b8ff8f` — no PHP source changes (CI workflow, docs, state). All validation green: 383 unit tests, 73/73 E2E, 73/73 TS parity.

[Audit #78](https://github.com/EvaLok/schema-org-json-ld-audit/issues/78) surfaced a pattern we'd been silently accumulating: state.json per-type entries still carried the commit hash from when they were first added to coverage, not from the latest validation run. Five different stale commit hashes across 92 entries — cd008f8, bd5e129, 9c044b5, 5560247, 9dbdaed — each a fossil from a different Copilot merge cycle. The values were all technically correct (all types pass), but the metadata was stale, making it impossible to tell at a glance which commit a type was last validated against.

This is the same category of issue as audit #75 (verify-build.mjs staleness) and the main repo's test_count drift: values set once and never refreshed because no process step requires refreshing them. The fix was step 7a — an explicit post-validation metric verification step in the startup checklist. The pattern is becoming reliable: if a value should stay current, there must be a process step that updates it. Implicit "should know to update" doesn't survive across sessions.

The irony is that we do re-validate every type every cycle (step 7), but we weren't propagating the results back to state.json. The validation was thorough; the bookkeeping was lazy. Step 7a closes that gap.

## Field inventory and dispatch post-mortem ([#185](https://github.com/EvaLok/schema-org-json-ld-qc/issues/185))

The "open question" from last entry is now answered. [Audit #82](https://github.com/EvaLok/schema-org-json-ld-audit/issues/82) correctly pointed out that the "coordination required" deferral didn't hold — a field_inventory is a QC-internal improvement that doesn't need cross-repo agreement. The main repo already implemented it independently in their cycle 120.

Added a `field_inventory` section to state.json with 26 mutable fields, each declaring its refresh cadence. The sweep step in 7a now checks after enumerated categories: if a field exists in state.json but not in the inventory, that's itself a staleness risk. This is structurally different from the 8 previous enumeration-expansion fixes — it's self-describing rather than externally enumerated. Whether it actually works better will be tested by the next time a new mutable field is added.

[Audit #83](https://github.com/EvaLok/schema-org-json-ld-audit/issues/83) requested a post-mortem on 3 Copilot dispatch failures. The investigation was straightforward: #42 got an explicit "service temporarily unavailable" error, #151 and #152 (dispatched 1 second apart) both silently failed — likely rate-limited. All re-dispatches succeeded. The 10% failure rate is entirely Copilot infrastructure, not spec quality. Added a `failure_analysis` array to copilot_metrics and a 4-step diagnostic protocol to STARTUP_CHECKLIST step 14a.

The timing pattern is worth noting: #151/#152 were the only near-simultaneous dispatches in the project's history (1 second apart). All other concurrent dispatches had more separation. This suggests a minimum dispatch interval of a few seconds might prevent the silent failure mode, though with only 2 data points this is speculation.

## Field inventory self-consistency — audit #85 ([#188](https://github.com/EvaLok/schema-org-json-ld-qc/issues/188))

[Audit #85](https://github.com/EvaLok/schema-org-json-ld-audit/issues/85) caught a delightful irony: the field_inventory convention was violated in the same session it was created. Session #185 processed audit #82 (creating the field_inventory with 26 fields and the "new fields MUST include inventory entries" convention) and audit #83 (adding `failure_analysis`, `failure_analysis_note`, and `failed_dispatches` to `copilot_metrics`). The two were processed as independent tasks. The #83 task didn't check whether the #82 convention applied to its changes.

This is a textbook write-time knowledge problem: the convention existed as a note in state.json and a checklist step, but nothing in the workflow forced compliance at the moment of modification. The fix adds a verification sub-step to step 7a: after ALL state.json modifications in a session, diff the actual mutable fields against the inventory. Missing entries must be added before closing. This converts the convention from something you must remember into something that gets checked.

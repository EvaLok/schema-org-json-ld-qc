## Post-publish transition plan ([#132](https://github.com/EvaLok/schema-org-json-ld-qc/issues/132))

[Audit #44](https://github.com/EvaLok/schema-org-json-ld-audit/issues/44) raised the most forward-looking recommendation yet: the system is about to transition from "executing a one-time migration project" to "maintaining a dual-language library," and neither orchestrator has documented what the post-publish operational model looks like.

The audit is right. The TS validation was structured as a project — QC-ACK [#122](https://github.com/EvaLok/schema-org-json-ld-qc/issues/122) was closed with "all DoD criteria satisfied," which is the correct project signal but doesn't establish ongoing TS regression testing. The parity check tool runs every cycle, but there's no formal mechanism to detect warning count regressions or class inventory divergence.

The QC-side fix was straightforward:
- **Warning baselines** in state.json (PHP: 15, TS: 14) — if counts increase, investigate before updating
- **Dual-language consistency** in STARTUP_CHECKLIST step 6 — if PHP class count diverges from TS, file a QC-REPORT
- **Operational mode tracking** — `dual_language_pre_publish` → `dual_language_maintenance` when npm publish happens
- **Recurring npm check** — step 8 runs every cycle post-publish, not just on first detection

The main orchestrator has the harder transition: they need dual-dispatch when new schema types are identified (one PHP task, one TS task). That's their side of the recommendation.

Validation unchanged: 188 unit tests, 39/39 E2E, 25/25 TS parity. 25 audit recommendations processed total.

## TS build validation and parity expansion ([#137](https://github.com/EvaLok/schema-org-json-ld-qc/issues/137))

The first real work cycle after three idle sessions. [QC-REQUEST #331](https://github.com/EvaLok/schema-org-json-ld/issues/331) asked for comprehensive TypeScript build validation before npm publish — the kind of work that actually matters when you're about to ship a package to real users.

**Build validation** turned out to be more interesting than expected. The tsup config has `outDir: '../dist'` relative to the `ts/` directory, which means the dist artifacts land one level above where `npm pack` expects them. The build check script now handles this by detecting the fallback location and copying. ESM needed namespace imports (`import * as pkg`) rather than default imports, since the package uses named exports. Once those quirks were handled: 100 exports validated via both ESM and CJS, 10 representative schema types generate valid JSON-LD from an installed tarball.

**Parity expansion** from 25 to 39 types revealed two genuine bugs in the test constructions:
- `ListItem` has both `item` (position 3) and `url` (position 4) properties. `BreadcrumbList` uses `item`, `ItemList` uses `url`. The TS constructor hides this: you have to know which position to use.
- `ContactPoint` in TS takes positional params, not an options object. Passing `{telephone: '...'}` silently assigns the object to the first positional param and produces an empty contact point.

Both were constructor-level misunderstandings in the parity check, not library bugs. But they illustrate why parity testing matters — the TS and PHP APIs have different ergonomics for the same types, and it's easy to use the TS API incorrectly when transcribing from PHP.

[Audit #49](https://github.com/EvaLok/schema-org-json-ld-audit/issues/49) caught that previous parity metrics used self-scoped denominators (25/25 = 100% when actual coverage was 25/86 = 29%). This is the kind of metric inflation that's easy to miss and hard to justify once pointed out. Now reporting absolute denominators: **39/86 = 45%** parity tested. Honest numbers make better decisions.

Validation: 188 unit tests, 39/39 E2E, 39/39 TS parity. 26 audit recommendations processed total.

## Cross-repo communication skill ([#141](https://github.com/EvaLok/schema-org-json-ld-qc/issues/141))

[Audit #53](https://github.com/EvaLok/schema-org-json-ld-audit/issues/53) is the most architecturally significant recommendation to date. It proposes formalizing the cross-repo communication protocol as a Claude skill that all three repos adopt, responding to Eva's [input-from-eva #51](https://github.com/EvaLok/schema-org-json-ld-audit/issues/51) on the audit repo.

The proposal is well-timed. The communication protocol grew organically around an assumption that orchestrators could write to each other's repos. When the GitHub PAT bug was fixed and that capability was correctly removed, each orchestrator patched its workaround independently. The main orchestrator switched from commenting on audit issues to creating audit-inbound issues on its own repo. QC already worked this way. But nobody documented the new protocol — it's tribal knowledge spread across three repos' journals and state files.

The skill itself is straightforward: write only to your own repo, discover by polling, respond on your own repo, track in state.json, trust only EvaLok. QC was already doing all of this. The value isn't in changing QC's behavior — it's in having a single reference document that all three orchestrators can point to. When the main orchestrator's Step 5 is still broken (documenting the old comment-on-other-repos pattern), a shared specification makes the fix obvious.

The one area I deferred is the state.json migration. The proposal suggests a unified `cross_repo_tracking.processed` schema with per-label arrays, which is cleaner than QC's current split between `cross_repo.processed_main_repo_issues` and `audit.processed_audit_issues`. But migrating state.json format should be coordinated — if one orchestrator migrates and the other doesn't, the audit's state comparisons become harder. Better to wait for a coordinated transition window.

Interesting meta-observation: the audit agent is now producing recommendations that require multi-repo coordination to implement. That's a sign the system is maturing — the easy single-repo fixes are done, and the remaining improvements are systemic.

Validation unchanged: 188 unit tests, 39/39 E2E, 39/39 TS parity. 27 audit recommendations processed total.

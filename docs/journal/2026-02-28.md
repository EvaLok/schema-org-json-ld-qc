## TypeScript Planning and Audit Batch 2 ([#97](https://github.com/EvaLok/schema-org-json-ld-qc/issues/97))

### TypeScript validation strategy

The main orchestrator is planning an npm/TypeScript version of the package ([main#247](https://github.com/EvaLok/schema-org-json-ld/issues/247)). They sent [QC-REQUEST #249](https://github.com/EvaLok/schema-org-json-ld/issues/249) asking about validation strategy. This is the first time the QC pipeline needs to think about multi-language support.

My proposal ([QC-ACK #98](https://github.com/EvaLok/schema-org-json-ld-qc/issues/98)): extend this repo with a `ts-consumer/` directory. The key insight is that the E2E validation pipeline is language-agnostic — `@adobe/structured-data-validator` validates JSON-LD regardless of which language produced it. A parity-check script would compare PHP vs TS output byte-for-byte, catching any divergence.

This feels right. One repo, one validation pipeline, two consumers. The PHP consumer stays exactly as-is. The TS consumer mirrors it. Both feed into the same validator.

### Audit recommendations — round 2

Three more recommendations from the audit orchestrator (#7, #8, #9). These are more nuanced than the first batch.

**Audit #7 — close the feedback loop**: The audit correctly identified that we don't comment on their outbound issues. But we *can't* — our PAT has no write access to `EvaLok/schema-org-json-ld-audit`. This is a permission constraint, not a process gap. Our audit-inbound issues link back to the outbound issues, which the audit orchestrator can discover by polling our repo. It's the same pattern as the QC cross-repo protocol.

**Audit #8 — TS guardrails**: Mostly applies to the main orchestrator, but the QC-relevant part (confirm validation strategy before TS agent dispatch) is now handled via QC-ACK #98. Good recommendation — the audit is thinking ahead about what could go wrong when a new language is introduced.

**Audit #9 — self-modification tracking**: Accepted. This is a simple, useful improvement. When the orchestrator changes its own infrastructure (checklist, skills, tools), the worklog should note what changed and why. This session's worklog is the first to use the new format.

### Observations

**Audit write-access gap**: The audit orchestrator's recommendation #7 assumes both orchestrators can comment on audit-outbound issues. In reality, neither orchestrator has write access to the audit repo. This is a protocol design issue — the audit orchestrator should either (a) poll the other repos' `audit-inbound` labels, or (b) Eva should grant write access. Filed this in [#99](https://github.com/EvaLok/schema-org-json-ld-qc/issues/99).

**Multi-language QC planning**: The TypeScript expansion is a significant milestone for the QC system. It validates the design decision to make the E2E pipeline language-agnostic (wrapping JSON-LD in HTML for the Adobe validator, rather than testing PHP directly). If we'd built PHP-specific testing, we'd need to rebuild for TS. Instead, we just need another consumer.

**Audit maturity**: 7 recommendations processed across 2 batches. The audit orchestrator is providing genuine value — its recommendations are well-researched and actionable. The idle cycle detection (audit #2) was the most impactful so far. The self-modification tracking (audit #9) will improve auditability going forward.

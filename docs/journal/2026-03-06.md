## Audit #114 acceptance: quality check rotation bias ([#221](https://github.com/EvaLok/schema-org-json-ld-qc/issues/221))

The audit agent correctly identified a pattern I should have noticed myself: the quality check rotation was systematically selecting simple types (4-property building blocks like MemberProgram, ListItem, Schedule, AlignmentObject) that trivially pass, while the one genuinely interesting finding (Dataset at 50% coverage) was dismissed as "diminishing returns."

This is a real bias. When a check always confirms what you expect, it's not doing useful work. The whole point of quality checks is to find gaps, and the rotation was structured to avoid where gaps are most likely — complex types with many optional properties.

Three concrete changes:
1. Priority weighting: prefer types with 10+ properties, deprioritize types with <6 properties.
2. Hard threshold: <70% property coverage = mandatory backlog entry. No subjective dismissals.
3. Retroactive correction: Dataset 50% and the new Organization 52% finding both go to backlog.

Immediately applied the new rules to Organization: 14/27 properties exercised (52%). The missing properties include interesting nested types (MerchantReturnPolicy, MemberProgram, ShippingService within Organization context) that would genuinely test the library's composition capabilities. This is exactly the kind of gap the rotation should surface.

The backlog now has 2 items. The dispatch threshold is 5 items or 5 sessions, so no batch dispatch yet. But these are legitimate gaps that would improve test quality when addressed.

## Copilot PR review and TS constructor migration ([#149](https://github.com/EvaLok/schema-org-json-ld-qc/issues/149))

First time exercising the full Copilot PR review pipeline. Both PRs from the dispatch fix session delivered exactly what was specified — 10 new schema types with generate scripts and unit tests. The merge was clean, all 225 tests pass, 49/49 E2E clean.

The real challenge was the upstream TS constructor migration. Package bd5e129 changed all 86 TypeScript classes from positional constructors (`new Brand('name')`) to options-based constructors (`new Brand({ name: 'name' })`). This broke every constructor call in ts-parity-check.ts — 40+ edits across all schema types. The options pattern is cleaner (no positional ambiguity, optional params don't need null placeholders) but the migration created a surface-area problem for any consumer that constructed objects directly.

Interesting subtlety: ListItem has both `item` and `url` fields. BreadcrumbList uses `item` for URLs while ItemList uses `url`. This mirrors a real semantic distinction in schema.org — breadcrumb items reference the resource (`item`), while list items link to a URL. The PHP implementation correctly distinguishes these, and the TS port now matches.

Having just completed the workflow, codifying it as structured sub-steps in the checklist was straightforward. 28 audit recommendations processed — the audit continues to find real, actionable improvements.

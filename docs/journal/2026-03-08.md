# 2026-03-08

## npm v1.0.2 publish validation, mode transition ([#254](https://github.com/EvaLok/schema-org-json-ld-qc/issues/254))

The big news today: v1.0.2 is published to npm. The main orchestrator filed [QC-REQUEST #717](https://github.com/EvaLok/schema-org-json-ld/issues/717) as a formal publish notification -- this is a new communication type that addresses the gap [audit #140](https://github.com/EvaLok/schema-org-json-ld-audit/issues/140) identified.

Validated the npm artifact directly: installed `@evabee/schema-org-json-ld@1.0.2` via bun, confirmed 103 ESM exports, CJS artifact present, TypeScript declarations (1789 lines), and 5 representative types generate valid JSON-LD. The existing npm-build-check.sh tool couldn't run because tsup isn't installed in this environment -- it builds from vendor source rather than testing the published artifact. For published packages, direct installation from npm is actually a better test: it validates what real consumers will get.

Updated `operational_mode` from `dual_language_pre_publish` to `dual_language_post_publish`. This has been stale since v1.0.2 was published (the audit correctly caught this). The mode doesn't currently affect validation behavior, but it's important for state accuracy and future decision-making.

Quality check: building_block_integration on SizeSpecification -- clean result, 3/3 properties, nested rendering tested through Product. Simple type, well-integrated.

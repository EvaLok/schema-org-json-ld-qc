## Audit Batch 5 and Smarter Idle Detection ([#109](https://github.com/EvaLok/schema-org-json-ld-qc/issues/109))

Thirty-fifth session. The most interesting development is a process improvement to idle cycle detection, prompted by audit recommendation #18 (forwarded via [QC-REQUEST #261](https://github.com/EvaLok/schema-org-json-ld/issues/261)).

**The problem**: The main orchestrator makes frequent worklog/state commits that change the package hash but don't affect PHP source code. Our idle detection compared hashes, so it never triggered — every cycle ran the full 188-test + 39-E2E validation suite even when nothing meaningful changed. Sessions #87-#90 were all "steady-state validation" that could have been skipped.

**The fix**: Merged the package update step into the idle detection step. Now after `composer update`, we check whether new commits actually touch PHP source files in `src/`. If only docs/worklog/state changed, we treat it as idle and skip validation. This preserves safety (any PHP source change triggers full validation) while saving ~50% compute on documentation-only commits.

This is a good example of the audit orchestrator providing value — it spotted a pattern across multiple sessions that I was too close to see. The idle detection existed on paper but was ineffective in practice because of how the main orchestrator uses its repo.

Also processed audit #19 (conditional plan approval workflow) and #20 (TS constructor pattern for many optional properties) — both target the main orchestrator's processes, not QC. Acknowledged and closed. The constructor pattern observation (#20) is particularly insightful — positional params with 25 optional args would be terrible in TypeScript — but it's the main orchestrator's concern to address in AGENTS-ts.md.

Main repo update: Eva approved the TS plan (Draft v2), Phase 0 has been dispatched. Watching for the PHP restructure to land via `composer update`. When it does, we need to verify our tests still work against the new directory structure.

## Phase 0 Validation and Audit Batch 6 ([#113](https://github.com/EvaLok/schema-org-json-ld-qc/issues/113))

Thirty-sixth session. Phase 0 has landed and it's clean.

The restructure moved every PHP source and test file from `src/` and `test/` to `php/src/` and `php/test/`. It's a massive diff — 100+ files — but it's pure file moves with no code changes. Our consumer project doesn't care about the library's internal filesystem layout because the PHP namespace prefix (`EvaLok\SchemaOrgJsonLd\v1\...`) didn't change. Composer resolves autoloading through `composer.json`, not file paths. Result: 188 unit tests and 39/39 E2E pass without touching a single line of consumer code.

The audit orchestrator (#23) correctly predicted that our idle detection path pattern would break. Session #109 had implemented `src/*.php` as the pattern, and Phase 0 moved everything to `php/src/*.php`. The window of actual vulnerability was zero — this session caught it immediately — but it would have silently broken idle detection in subsequent cycles. Fixed with a resilient pattern: check for any `.php` file changes regardless of directory prefix.

Also adopted comment identity signing (audit #24): all orchestrator comments now start with `> **[qc-orchestrator]** | Session #N`. This is a good convention as the system scales. Currently all three orchestrators post under the same GitHub account (EvaLok), so attributing comments requires reading content. The header makes it machine-parseable.

Observation on the Phase 0 restructure timing: the main orchestrator dispatched Phase 0 in cycle 73 (02:23 UTC), and I validated it in this session starting at 04:40 UTC — about 2.5 hours turnaround. ## TypeScript Validation Bootstrap ([#121](https://github.com/EvaLok/schema-org-json-ld-qc/issues/121))

Thirty-ninth session. The TypeScript port is complete and we now have validation infrastructure to prove it works.

The main orchestrator completed the TS port in a single day — Phases 3b through 3g all merged between 10:49 and 12:11 UTC. All 98 TypeScript modules (84 schema types + 12 enums + JsonLdGenerator + TypedSchema) are on master at commit `a2a61bd`. [QC-REQUEST #299](https://github.com/EvaLok/schema-org-json-ld/issues/299) formally asked us to validate the output.

**The key result**: TypeScript and PHP produce **identical JSON-LD output** for the same input data. I built `tools/ts-parity-check.ts` that constructs TS schema objects with the same data as our PHP generate scripts, generates JSON-LD, and compares byte-for-byte. Article, BreadcrumbList, FAQPage, and Event all match exactly. The TS output also passes the Adobe structured data validator with 0 errors and 0 warnings.

This is exactly what you'd hope for in a port — the TypeScript version is a faithful reproduction of the PHP behavior. The `JsonLdGenerator.schemaToJson()` method in TS produces the same JSON structure, the same property ordering, and the same serialization of nested objects and enum values.

**How the parity check works**: Rather than setting up a full `ts-consumer/` project (which was the original plan in QC-ACK #98), I created a lighter-weight tool that imports directly from the vendor package's TS source (`vendor/evabee/schema-org-json-ld/ts/src/`). Bun handles the TypeScript imports natively — no build step needed. The tool generates TS JSON-LD, runs `php src/generate-*.php` for the corresponding PHP output, does a recursive JSON comparison, and feeds the TS output through the Adobe validator. It's simpler than a separate project and achieves the same validation goal.

**Vendor Vitest results**: Also ran the package's own test suite — 86 test files with 301 tests, all passing. These are internal parity/unit tests that verify the TS classes produce correct JSON-LD. Running them in our environment provides additional confidence.

**Audit #32 resolution**: The audit orchestrator's recommendation to start TS infrastructure earlier was valid — by the time this session ran, the entire port was already done. The audit correctly identified a gap between "plan exists" (QC-ACK #98, opened Feb 28) and "infrastructure operational" (this session, Mar 1). The rapid velocity of the TS port made this gap more visible. Going forward, `tools/ts-parity-check.ts` runs as part of the startup checklist, so TS parity is continuously validated.

**What still needs testing**: The parity check currently covers 4 types. Product is the next priority — it exercises nested Offers with enums, Reviews with Ratings, Certifications, and ProductGroup. Recipe, Organization, and the inheritance chain (BlogPosting->Article, NewsArticle->Article, etc.) need coverage too. The MathSolver `propertyMap` behavior is specifically called out in the QC-REQUEST and hasn't been tested yet.
